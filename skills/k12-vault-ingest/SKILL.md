# K12 Vault Ingest

## What this does and where it sits

The Thrive K12 knowledge vault has two axes: **chronological digests** (the weekly synthesis
output, append-only raw material) and **accreting entity & theme pages** (the durable knowledge,
built up over time so longitudinal questions can be answered from a pre-synthesized page instead of
re-derived). This skill performs the weekly hand-off between them: it takes one weekly digest and
executes the update procedure — file the digest, grow every entity and theme page it touches,
refresh the index, keep the vocabulary consolidated.

Two skills, one pipeline: **`k12-synthesis` produces the digest; this skill files it and accretes
knowledge from it.** This skill never fetches news or invents facts — it only moves what the digest
already established into its durable home, with provenance intact.

## The cardinal rule: no fact without provenance, no fact the digest didn't state

Everything written into the vault must trace back to a specific item in the digest, and every
Timeline entry must carry that item's source link. If the digest didn't say it, it does not go into
the vault — do not infer, extrapolate, or backfill from prior knowledge. The vault is a research
asset whose value is that a reader can trust and verify every fact. A plausible-but-unsourced
sentence is worse than an omission. When in doubt, leave it out.

## Before you start: read the vault's own rules

The vault is the source of truth for its own schema and vocabulary, and both will evolve. **Read
these two files at the start of every run** rather than relying on your memory of them:

- **`CONVENTIONS.md`** — the operating manual: the three front-matter schemas, the snapshot +
  Timeline body structure, the provenance rule, the markdown conventions, and the weekly update
  procedure. This skill executes that procedure; CONVENTIONS.md defines it.
- **`TAGS.md`** — the controlled vocabulary: the six layer slugs, the canonical tags, and the
  naming rules. You check tags and entity slugs against this before writing, and add new canonical
  terms here when genuinely needed.

If anything below appears to conflict with the vault's current CONVENTIONS.md, **CONVENTIONS.md
wins** — it is the live contract; this skill is the executor.

### Locate the vault

The vault root is the directory that contains `index.md` (with `type: index` in its front-matter)
alongside `CONVENTIONS.md` and `TAGS.md`, plus `digests/`, `entities/`, and `themes/`
subdirectories. It is named `thrive-k12-vault/` by convention. If the user gave a path, use it. If
you can identify exactly one such directory in the working tree, use it. If there are zero or more
than one, ask the user for the vault path rather than guessing — writing into the wrong tree is
hard to undo.

## Step 0 — Get the digest and confirm the window

Obtain the digest markdown, in priority order:

1. A path the user gave you (e.g. `digests/2026/2026-W26.md`, or a loose file from a synthesis run).
2. The digest just generated in this session by `k12-synthesis` (its output markdown).
3. If neither exists, stop and ask — this skill ingests a digest, it does not create one. Point the
   user at `k12-synthesis` to generate one first.

Read the digest. Its header carries the window (`### Week of {start} – {end}`). Derive:

- **`week`** — the ISO 8601 week (Mon–Sun) that the digest's end date falls in, formatted
  `YYYY-Www` (e.g. `2026-W26`). Compute it; don't guess. `date -d` / `python3 -c` with the
  `isocalendar()` of the end date is a reliable check.
- **`date_start` / `date_end`** — the window bounds from the header, as `YYYY-MM-DD`.
- **`generated`** — today's date, `YYYY-MM-DD`.

State the derived week back to the user so a wrong window is caught before anything is written.

## Step 1 — File the digest

The synthesis output is just the briefing body; it does **not** carry the vault's digest
front-matter. You add it. Save the digest to `digests/<YYYY>/<YYYY>-Www.md` and prepend the
front-matter defined in CONVENTIONS.md. Derive each field from the digest content:

- **`layers`** — only the layers that actually have items this week. The digest's section headings
  map to layer slugs (Federal Funding & Policy → `federal`, State Systems → `state`, District
  Purchasing Layer → `district`, Charter School Sector → `charter`, Technology & Vendor Layer →
  `vendor`, Ideas & Influence Layer → `ideas`). Confirm the mapping against TAGS.md.
- **`entities`** — canonical slugs of the entities named in the digest (see Step 2 for slugging).
- **`states`** — two-letter US postal codes the digest touches.
- **`tags`** — from TAGS.md only. If the week surfaces a genuinely new recurring topic, add the tag
  to TAGS.md first (Step 5), then use it here.
- **`items`** — the count of distinct items in the digest.

Keep the digest body verbatim — it is the append-only raw record. Don't rewrite it; you're filing
it, not editing it. If a digest for this week already exists, don't silently overwrite it: tell the
user and confirm whether this is a re-run (re-file) or a correction.

## Step 2 — Identify what the week touches, then grow each page

This is the judgment core. Read the digest as a set of items, and for each item decide what durable
knowledge it advances.

### Entities

An entity is a specific named actor: a **company** (vendor, e.g. a telehealth or SEL platform), an
**agency** (federal/state body), a **program** (a named grant program or initiative), or a **fund**.
For each entity the digest names:

1. **Resolve to a canonical slug.** Search existing pages under `entities/` and the aliases in
   TAGS.md / existing front-matter before creating anything. The same real-world entity must have
   exactly one page — match on aliases ("Hazel" → `hazel-health`), not just exact strings. Reuse the
   existing page; only create a new one if none matches.
2. **Pick the type and folder.** `company` → `entities/companies/`, `agency` → `entities/agencies/`,
   `program` → `entities/programs/`. `fund` lives with programs unless the vault has grown a
   dedicated folder. Set `entity_type` accordingly.
3. **Create from template if new.** Copy the matching `entities/<type>/_template.md`, fill the
   front-matter, set `first_seen` **and** `last_updated` to this week. Strip the template's
   placeholder comments.
4. **Update if it exists.** Never change `first_seen`. Bump `last_updated` to this week. Update
   `status` (active | acquired | shutdown | dormant), `aliases`, and `tags` only if the digest gives
   you reason to. Leave the rest.
5. **Overwrite the Current snapshot.** Rewrite it to state what is true about the entity *now*, in a
   few sentences, ending with `*(as of <YYYY-Www>)*`. This section is mutable current truth — the
   snapshot should read as if written today, folding in what the new item changes. The history lives
   in the Timeline, so the snapshot doesn't need to recite it.
6. **Prepend a Timeline entry**, newest first, in the CONVENTIONS.md format:
   ```
   ### <YYYY-Www> — short headline of what happened
   *Source · Month DD, YYYY* · [link](url)
   1–2 sentences, key facts raw (dollar amounts, names, dates).
   ```
   Keep the facts raw — dollar amounts, program names, agencies, dates — in the source's terms, not
   paraphrased into mush. **Every entry must have a source link.** If a digest item has no link, that
   fact cannot enter a page; keep it in the digest only.

   **Copy the link, never retype it.** The link must be the *exact* URL from the matching digest
   item. A URL is an opaque string — if you reproduce it from memory or "tidy it up," you will
   eventually write a plausible-but-wrong link that looks fine and fools every eyeball check. So
   don't type it: pull the exact URL out of the digest and paste it. A reliable way to get the right
   links is to grep the digest for the item, e.g.
   `grep -n "ETS acquires" digests/2026/2026-W27.md` then copy the URL from the `[link](...)` on
   that source line. Step 4 runs `check_links.py`, which fails the run if any link you wrote for
   this week isn't present verbatim in the digest — so a mistyped link cannot slip through.

### Themes

A theme is a *cross-cutting arc*, broader than any single entity: an ongoing storyline the vault
tracks over months (e.g. `school-medicaid-mental-health`, an SEL-funding arc, the ESSER cliff). Not
every item spawns a theme — create or update a theme page when the week **advances an arc worth
answering longitudinal questions about**. Prefer updating an existing theme over minting a
near-duplicate; check `themes/` and TAGS.md first.

For each theme the week advances: create from `themes/_template.md` if new (set `last_updated`,
`related_entities`, `related_states`, `tags`), otherwise update `last_updated` and extend
`related_entities` / `related_states` / `tags` as the arc grows. Then **overwrite the Current
snapshot** to the synthesized current state of the arc, and **prepend a sourced Timeline entry** —
same rules as entities. A theme's snapshot is where you synthesize across the entries; keep it tight
and current, and keep every underlying fact sourced in the Timeline.

### Re-run discipline

If a page already has a Timeline entry for this same week (you're re-running or correcting), update
that entry in place rather than adding a second one for the same week — the Timeline is one entry
per event, newest first, not an append-on-every-run log.

## Step 3 — Refresh index.md

`index.md` is the agent's router; its three dynamic sections must reflect the vault after this run.
Update, keeping the descriptive prose above them intact:

- **Active themes** — one line per theme page (link + a few words), from `themes/`.
- **Tracked entities** — grouped under Companies / Agencies / Programs, one line per entity page
  (link + a few words), from `entities/`.
- **Digests by week** — one line per digest, **newest first**, linking `digests/<YYYY>/<YYYY>-Www.md`.

Replace the "populated as weeks are added." placeholders as sections gain content. Set the index's
`last_updated` front-matter to this week. Rebuild these sections by scanning the actual files and
their front-matter — don't rely on what you remember the index said; make it match the tree on disk.

## Step 4 — Validate before you finish

**First, run the link-provenance gate — this is not optional.** It is the one check that catches a
hallucinated or mistyped source link, which is the highest-stakes failure this vault can have (a
plausible-but-wrong URL looks correct and defeats manual review):

```
python3 check_links.py <vault_root> digests/<YYYY>/<YYYY>-Www.md
```

`check_links.py` (bundled with this skill) verifies that every link inside this week's `### <week>`
Timeline entries appears **verbatim** in the digest. If it reports a violation, you retyped a link
instead of copying it — fix it by pasting the exact URL from the digest item, then re-run until it
passes. Do not proceed while it fails.

Then do a read-back pass and fix anything else that fails — the vault's worth depends on these holding:

- **Provenance:** every Timeline entry you wrote (entities and themes) has a real source link, and
  `check_links.py` passes. No unsourced facts anywhere on a page.
- **No fabrication:** every fact traces to a digest item. Nothing inferred or imported from outside
  the digest.
- **Vocabulary:** every tag used is in TAGS.md; every layer slug is one of the six; every filename is
  a lowercase-hyphenated slug; no duplicate page for an entity/theme that already existed under
  another name.
- **Front-matter:** digest, entity, and theme front-matter match the CONVENTIONS.md schemas;
  `first_seen` was preserved on existing entities; `last_updated` bumped everywhere touched.
- **Markdown conventions (clean .docx conversion):** ATX headings; standard bullets and
  `**bold**`/`*italic*` only; plain `[text](url)` links; **never a link nested inside an emphasis
  span** — close the italic before the link (`*Source · June 28, 2026* · [link](url)`, not
  `*Source · … [link](url)*`); no raw HTML, no nested tables.

## Step 5 — Reconcile the vocabulary

If the week introduced a genuinely new recurring topic that deserves a canonical tag, add it to
TAGS.md (with its one-line description) as part of this run, so the vocabulary stays consolidated and
the next run reuses it instead of coining a synonym. Don't add one-off tags that won't recur — the
list should stay short and sensible. New entity *slugs* live on their pages; new canonical *tags*
and any new alias conventions belong in TAGS.md.

## Step 6 — Report, and offer to commit

Summarize what you did: the digest filed (path + week), each entity and theme created vs. updated,
any new tags added, and anything you deliberately left out (e.g. an item with no source link, or a
marginal entity you chose not to track) with the reason. Surface any validation issue you couldn't
cleanly resolve rather than burying it.

The vault is versioned in git. Offer to stage and commit the run with a clear message (e.g.
`ingest digest 2026-W26 into vault`), but don't commit unless the user wants it.

## Style notes

- Match the digest's register: facts raw and verifiable, synthesis thin and only where it earns its
  place. The snapshot is the one spot you synthesize; the Timeline stays factual.
- Prefer updating over creating. A vault that fragments one entity across three near-duplicate slugs
  is worse than one with slightly coarser pages. Consolidation is a feature.
- Terse weeks are fine. If the digest is light, the vault update is light. Don't manufacture entities
  or themes to look busy.
